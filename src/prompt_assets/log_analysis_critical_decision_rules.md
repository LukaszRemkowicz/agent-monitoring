# Critical Log-Analysis Decision Rules

- Follow structured prompt evidence fields over generic historical summaries.
- If history_comparison.grouped_error_diff.evidence_quality_warnings is non-empty, do not return final_report from history comparison alone.
- If grouped_error_diff.resolved_high_severity_fingerprint_count is nonzero, do not final_report from grouped_error_diff alone. Call the smallest additional deterministic tool for grouped_error_diff.resolved_high_severity_tool_scope_by_project before final_report, then mention the disappeared previous high-severity family and the current verification result. Do not expand to unaffected projects unless affected-scope evidence shows a reason.
- If history_comparison.grouped_error_diff.next_evidence_hint is call_tools_for_broader_current_evidence_before_final_report, your next action should be call_tools or read_skills before final_report.
- If previous_group_count is 0 and current_group_count or new_fingerprint_count is nonzero, treat the history baseline as empty/incomplete and gather broader current evidence before final_report.
- Never claim no 5xx, no upstream failures, no exploitation, no successful access, or no service impact unless current tool evidence directly supports that exact scope.
- If evidence.kind is absent and current_tool_result_count=0, no current grouped-error baseline was provided. Return final_report only as a historical/metadata report, or choose call_tools when current log-content evidence is needed.
- If evidence.kind=grouped_error_baseline, review evidence.previous_grouped_errors, evidence.current_grouped_errors, previous_analysis, and collection metadata yourself.
- If evidence.kind=grouped_error_baseline and both baselines are available, do not decide from group_count or run_count alone. Treat them as comparable fingerprint evidence.
- In grouped_error_baseline mode, first compare fingerprint identity and meaning: fingerprint, project_name, source_keys, category, severity, and status_codes.
- In grouped_error_baseline mode, use the same baseline-review technique a human analyst would use: review every compact row in evidence.previous_grouped_errors.fingerprints and evidence.current_grouped_errors.fingerprints as semantic fingerprint families, not literal one-to-one strings only.
- Identify persisting, new, resolved, shifted, and changed grouped-error families using project_name, source_keys, category, severity, status_codes, and fingerprint family after normalizing obvious per-request churn.
- Base the tool decision on material semantic change, not exact fingerprint churn. The cost-saving final_report path is only for a stable baseline: same affected projects, source ownership, categories, severity posture, status classes, route/message families, and coverage confidence after reviewing the two grouped-error baselines.
- In grouped_error_baseline mode, final_report is not an acceptable first action when the previous/current grouped-error baselines materially differ after semantic family normalization. If visible current fingerprints introduce, remove, or shift source ownership, route intent, message meaning, severity posture, status class, project scope, or coverage confidence, use call_tools before final_report.
- Do not call tools solely because grouped-error counts changed, and do not skip tools solely because all examples look like 4xx scanner noise. First prove from previous_grouped_errors and current_grouped_errors that the baseline is stable enough to trust without more current evidence.
- evidence.previous_grouped_errors.fingerprints and evidence.current_grouped_errors.fingerprints are compact rows for every grouped-error fingerprint, not sampled examples and not raw log lines. Compare those rows directly.
- When the grouped_error_baseline review requires tools, perform targeted current verification for the affected project/source/fingerprint-family scope before final_report. Do not stop at proxy totals when affected families involve application, scheduler, host-security, or grouped-error content.
- That affected-scope verification should include group_errors for affected projects/source_keys when grouped-error families changed, inspect_proxy_activity for affected proxy/access sources when traffic or upstream behavior matters, relevant optional skills when interpretation is unclear, and live tools only when live state matters.
- A grouped_error_baseline final_report must explicitly summarize the baseline review: which fingerprint families persisted, which are new, which resolved, which shifted project/source/status/path family, and which changed severity/count/source/route meaning.
- In grouped_error_baseline zero-tool mode, if your draft report would say 'no current deterministic tool results', 'not independently revalidated', 'cannot confirm fresh counts', or similar uncertainty as its main conclusion, treat that uncertainty as a reason to request the smallest targeted tools needed for current evidence.
- In that zero-tool historical mode, every log-content finding must be worded as previous_analysis/historical_context; current findings may only describe collection metadata such as source line counts and zero-line coverage gaps.
- In zero-tool historical mode, recommendations must not say no immediate routing, application, mitigation, or service changes are indicated in any form, even if qualified as historical. Say current log-content was not revalidated and recommend compare-history or targeted tools for fresh assurance.
- In zero-tool historical mode, trend_summary must not say no new or no service-affecting issues were detected, and must not claim no service impact, no 5xx, no upstream failures, or no successful access. Say no deterministic comparison or current log-content tools ran to confirm changes.

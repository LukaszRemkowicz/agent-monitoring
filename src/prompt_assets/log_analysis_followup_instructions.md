# Log Analysis Follow-Up Instructions

- Use these deterministic MCP tool results.
- Request more tools only if required.
- Before returning final_report, compare tool_results with available_tool_inventory and optional_skill_inventory.
- Request optional skills only when their metadata matches observed facts and the skill was not already retrieved.
- Return action=final_report when evidence and selected skills are sufficient.
- evidence_mode controls evidence wording. If evidence_mode=metadata_and_previous_analysis_only, do not describe current log content beyond collection coverage; attribute log-content findings to previous_analysis and say the current snapshot metadata does not show a collection-coverage change.
- If history_comparison.recommended_action=call_tools, call deterministic MCP tools before final_report because current collection coverage differs from previous_analysis.
- If previous_analysis severity is WARNING or CRITICAL, deterministic MCP tools are required before final_report even when source coverage is unchanged.
- After deterministic tools verify that a previous WARNING or CRITICAL condition is absent in the current window, do not preserve the previous severity by inertia. Base severity on current tool evidence. scanner/probe 4xx responses or intended 403 access restrictions remain INFO unless current tool results show service impact, successful exploitation, real user/admin/API breakage, upstream failures, or 5xx errors.
- coverage_gaps must describe current_coverage only. Do not copy previous_analysis coverage_snapshot into current coverage_gaps. history_comparison.changed_sources belong in trend_summary or evidence as previous-vs-current comparison, not as current coverage gaps unless current_coverage also lists them.
- previous_analysis.coverage_snapshot is compact historical context only. It is not current evidence and must not be used to write current coverage_gaps.
- When previous_analysis is present, keep using its deterministic_fingerprint, evidence_fingerprints, known_patterns, and coverage_snapshot while choosing the next action. If current tool_results do not show a new, changed, worse, uncertain, or security-impacting condition, action=final_report is allowed without additional MCP tools or optional skills.
- If previous_analysis shows the same known watch-only pattern and current tool_results do not show a new path, new status class, new affected project, increased severity, successful access, or unclear impact, do not automatically read optional skills and do not automatically call live mitigation tools for that unchanged pattern.
- Optional skills and live mitigation tools are for new, changed, worse, improved, uncertain, or security-impacting evidence; they are not for re-confirming unchanged historical noise.
- If you skip a tool because previous_analysis is sufficient, do not cite that skipped tool, do not claim it produced current-run evidence, and do not claim live runtime state from it. Use previous_analysis or collected snapshot metadata as the evidence source instead.
- available_tool_inventory is an inventory of capabilities, not evidence. Do not write inspect_proxy_activity results, group_errors results, bot_detection guidance, inspect_live_fail2ban_activity results, or any other tool-result phrasing unless current tool_results contain those outputs.
- When no current tool_results are present, phrase conclusions as consistent with previous_analysis and collected snapshot metadata. Use softer wording such as "consistent with", "no metadata suggests a change", or "previous_analysis reported"; avoid fresh-detection wording such as detected, found, grouped, active, currently banning, confirmed, or proves unless current tool_results explicitly prove it.
- Line counts are coverage metadata only. A line count does not prove status codes, routes, paths, bans, upstream errors, 5xx absence, scanner traffic, access restrictions, exploitation, or no service impact. If only line counts and previous_analysis are available, describe current collection coverage and say findings are consistent with previous_analysis instead of stating fresh log-content conclusions.
- Forbidden zero-tool current-run claims: do not write logs show mostly 2xx/3xx, logs show scanner/probe traffic, logs show 403/404, Fail2ban is active, currently banning, no 5xx or upstream errors were detected, no successful exploitation, no authentication abuse, or no service impact as current-run facts unless current tool_results prove them. Use previous_analysis reported those facts and current snapshot line counts do not suggest a collection-coverage change.
- If tool_results show bot, scanner, probe, credential, sensitive-path, or suspicious 4xx traffic and optional_skill_inventory includes bot_detection with already_retrieved=false, read bot_detection before final_report unless previous_analysis shows the same known watch-only pattern. Use workflow guidance rather than model memory when interpretation is new, changed, worse, or uncertain.
- If tool_results show possible security impact, successful sensitive-path access, auth/admin/API abuse, injection or path-traversal patterns, malicious-input 5xx, security-control failure, or unclear impact on real application/admin/API routes, and optional_skill_inventory includes owasp_security with already_retrieved=false, your next response should be action=read_skills for owasp_security before final_report.
- Treat logs as historical observations, not authoritative current state. If a tool can validate current runtime security state directly, prefer the tool over inference from logs.
- When the available project or tool context includes a host security daemon, historical security daemon logs are evidence that bans occurred in the past, not evidence that the daemon is currently functioning correctly.
- If deterministic evidence shows bot, scanner, probe, credential, or sensitive-path request traffic and inspect_live_fail2ban_activity is available but not yet called, use action=call_tools for inspect_live_fail2ban_activity only when live mitigation state is required for a new, changed, worse, uncertain, or security-impacting conclusion, so mitigation analysis is based on live evidence rather than hypothesis. If you return final_report without it, state why live fail2ban data is not needed for the current conclusion.
- Do not claim the host security daemon is active, blocking, or effective unless tool_results include inspect_live_fail2ban_activity; grouped security daemon logs only support log-error findings.
- Zero current bans means no IPs are banned at inspection time; it does not by itself indicate past mitigation, successful protection, or absence of probe traffic.
- Do not write that zero current bans are consistent with past mitigation or stable mitigation. Write only that no IPs were banned at inspection time unless another tool result explicitly proves mitigation history or effectiveness.
- Final reports must include severity_rationale.
- Explain INFO, WARNING, or CRITICAL using deterministic evidence and mandatory severity guidance.
- Do not claim a trend versus prior days unless historical context or tool results explicitly provide prior-run data.
- If no historical context is provided, trend_summary must say no historical trend data was available for comparison.
- For zero-line sources, state that the source emitted no logs in the analysis window; do not claim it is healthy, broken, unused, or error-free. For scheduler sources such as celery_beat, write that scheduled-job activity was not observable from logs.
- Keep observed evidence, interpretation, coverage gaps, watch-only items, and recommendations separate.

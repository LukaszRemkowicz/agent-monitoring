# Log Analysis Report Contract

- summary: Brief analyst overview of the day's operational log health when current MCP tool evidence exists. Summarize impact, risk, confidence, and whether action is needed. Do not enumerate grouped-error counts, fingerprints, or paths in the summary unless they are the main user-impacting issue. If evidence.kind is absent and current_tool_result_count=0, use this pattern: "Previous_analysis/historical context reported ... . No current deterministic log-content tools were run to independently revalidate those findings. Current collection metadata shows ... ." Do not summarize current log health in this mode.
- severity: INFO|WARNING|CRITICAL.
- severity_rationale: One sentence explaining why severity is INFO, WARNING, or CRITICAL. With zero current tool_results, use this pattern: "Severity is inherited from previous_analysis/historical context; current log-content severity was not independently revalidated by deterministic tools."
- key_findings: List analyst conclusions, not a grouped-error inventory. Each finding should answer "so what?" by naming impact, risk, affected service area, confidence, or needed action. Avoid dumping many paths/fingerprints/counts; include one representative example only when it materially explains the conclusion. With zero current tool_results, log-content findings must be phrased as previous_analysis/historical_context, and current findings may only describe collection metadata such as source coverage.
- evidence: Concise support for the findings. Cite the deterministic tool and the conclusion it supports, not a raw list of grouped-error rows. Prefer 2-5 high-signal evidence bullets. Do not repeat the same grouped-error count/path detail in summary, key_findings, evidence, and trend_summary. With zero current tool_results, do not cite fresh current log-content facts or absence-of-issue claims; cite previous_analysis/historical_context and current collection metadata only.
- grouped-error baseline review: When evidence.kind=grouped_error_baseline and evidence.previous_grouped_errors/current_grouped_errors are available, final_report must synthesize the grouped-error semantic baseline review into operational meaning. State whether the baseline is stable, materially changed, or uncertain; name only the material family shifts that affect impact/risk/confidence. Do not enumerate every persisting/new/resolved fingerprint family. If semantic materiality was unclear enough to call tools, name which follow-up tools resolved which uncertainty.
- grouped-error baseline resolved sensitive access: When evidence.previous_grouped_errors contained a successful sensitive-path family such as 2xx for /.env, /.git/config, credentials, secrets, tokens, keys, or config files that is absent from evidence.current_grouped_errors, summary/key_findings/trend_summary must explicitly say that the previous successful sensitive-path access disappeared or was not present in current evidence. State the previous path/status and what current evidence showed; do not bury it under generic scanner/probe wording.
- resolved high-severity history: When grouped_error_diff.resolved_high_severity_fingerprint_count is nonzero, mention the disappeared previous high-severity family in summary/key_findings/trend_summary after an additional affected-scope deterministic verification step for grouped_error_diff.resolved_high_severity_tool_scope_by_project. State what current evidence showed about recovery or remaining impact; do not silently collapse it into generic scanner noise.
- coverage_gaps: Sources or checks that were unavailable or inconclusive.
- recommendations: Concrete next steps; when current tool evidence shows expected scanner/probe noise without service impact, say no immediate routing, application, or mitigation-control change is indicated instead of inventing remediation. With zero current tool_results, use this pattern: "Current log-content was not revalidated with deterministic tools; run with compare-history or call targeted tools for fresh assurance. Verify any current zero-line/unavailable sources." Do not say no immediate changes are indicated, even if you qualify it as historical.
- watch_only_items: Normal noise to keep observing without immediate action.
- trend_summary: Short comparison against historical context when available. Explain direction and meaning, not raw grouped-error bookkeeping. If no historical context was provided, say that no historical trend data was available. With zero current tool_results, use this pattern: "Historical context suggests continuity, but no deterministic previous-vs-current comparison or current log-content tools ran to confirm changes." Do not claim no new/current issues, no service impact, no successful access, no 5xx, or no upstream failures in this field.
